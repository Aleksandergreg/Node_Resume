Git Documentation

Essential commands and best practices for version control

Git is a version control system that, cut down to the bone, helps us as developers to keep track of the changes we made to our code over time. Imagine it like this, you want to make a change to your project, but you are afraid it will make your project break, so you save your project as Project_v1, so you are sure you have a working version. Then you implement your new feature, so now you save your project as Project_v2. This would solve the issue, but it quickly gets messy, think of you having projects called Project_final and Project_final_final and so on.

It also allows us to collaborate and share our code with group members, and other people who would be interested in our code. Because of the version control it also allows us to play around with new features, without having to be afraid of breaking our application, as we can make roll back to earlier commits in the projects, create branches, so we don't break the code that exists on either our development or production branch.

The most important Git commands to be able to work with Git on GitHub or any other cloud-based platform for version control and collaboration, using any kind of terminal:

1 Cloning a repository 

git clone <url-to-the-repository-you-would-like-to-work-or-collaborate-on>

2 Initializing Git

If your project is not already a Git repository (not needed if you have cloned)

git init

3 Staging changes 

git add .

git add -A
git add .
will stage all your new and modified files, but does not stage deletions, in your current directory (i.e., the directory you have cd'ed into).
git add -A
will stage all the changes you have made to your project, including deletions, no matter which path you are on in your project.

4 Creating a snapshot of your changes

With the git add command, we have actually not saved the changes anywhere—we have just told Git that it is these changes that we would like to continue with. To create a snapshot of your changes, use this command: 

git commit -m "a good commit message here"

Commit Message Best Practices:

Making good commit messages is really important, but can also be a little tricky. You do not want to write 1500 words here—about everything you changed in great detail (that should be added as documentation if necessary). But also, you don't want to create commit messages that simply say "fixed stuff."

Think about doing it in keywords that briefly describe what you have implemented or changed, in this manner:

[Type], Short description (Optional File or module name)

Example: "fix: resolve null pointer in bookingController"

It is also important to stage and commit changes each time you make a meaningful change, so a single commit does not contain hundreds of lines of unrelated code. By following this practice, it's easier to find specific changes later without sifting through unnecessary details.

5 Connecting to a GitHub repository

With the git add and git commit commands, we have now saved our changes to our local git history. Next, we'd like to push these changes to a remote repository for an overall better view and to collaborate with others. (This step is not necessary if you used git clone, as your local repository is already linked.) 

git remote add origin https://github.com/your-username/your-repository.git

If you want to check the remote URL—and not just assume it's linked—use: 

git remote -v

6 Pushing changes to GitHub 

git push -u origin main

This tells Git to push to your upstream branch called main and set this as your default upstream branch. You can push to any branch by changing "main" to a different branch name, and of course adhering to your chosen branching strategy. Once you have done this, future pushes can be done with: 

git push

7 Pulling changes from GitHub 

git pull

As Git and GitHub are used for collaboration, the current structure of your project might have been changed by someone else. Before you start working on your own tasks, it's a good idea to first pull the latest changes down locally.

IMPORTANT

Certain things should not be pushed to Git, but just kept locally on your machine. For example, environment variables (often in a .env file) should never be pushed to it, as they can pose a security risk if leaked (e.g., usernames and passwords to services and databases, API keys). Also, items that are unique to your local environment (like OS-specific files or automatically generated folders such as node_modules) should be excluded, because these are not useful to others and can bloat repositories.

You can achieve this by adding these specific files and folders to your .gitignore file, which should be placed in your project's root directory. This ensures Git ignores them when staging and committing.

# Example .gitignore
node_modules/
.env
.DS_Store
.vscode/
*.log
dist/
